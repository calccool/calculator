Technical Justification (Process Mathematics)

The efficiency of local repacking can be expressed by comparing the processing time. In the classic cloud converter, the total time $T_{cloud}$ is the sum of the file upload time to the server $t_{up}$ , the server encoding time $t_{encode}$ and the download time $t_{down}$ :

$$T_{cloud} = t_{up} + t_{encode} + t_{down}$$

In our converter based on Fast Remux technology, network delays are completely eliminated. $T_{local}$ processing time depends solely on the read/write speed of your $V_{disk}$ drive and the $S$ file size:

$$T_{local} = \frac{S}{V_{disk}}$$

In this case, the bitrate (quality) of the final video $Q_{out}$ is always strictly equal to the bitrate of the original file $Q_{in}$ , since there is no mathematical loss function during remixing: $Q_{out} \equiv Q_{in}$ .

What is important to know about formats (Compatibility limitations)

MP4 and WebM are media containers (a kind of “box” for video). Inside MP4 there is usually video compressed with the H.264 or HEVC codec. The WebM format (developed by Google) has historically been designed for VP8 or VP9 codecs.

Our tool extracts the original H.264 stream and safely places it in a WebM container without re-encoding. Most modern browsers (Google Chrome, Edge, Opera) and modern video players (for example, VLC) understand this connection perfectly and play the file without problems. However, if you plan to feed the resulting file into professional editing programs with strict standards checks, you may need to do a full re-encoding with a codec change.
